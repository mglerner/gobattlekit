Cross-project contract assessment (the three questions asked)

(1) user-thresholds JSON schema — LIVE and BROKEN AT THE SEAM. Schema agreement is real: the emitter (pogo-simulator deep_dive_rendering.py, commit b97113f) produces exactly {species: {League: {name: {attack, defense, stamina[, ivs]}}}} with capitalized league labels, and gobattlekit's check_thresholds + load_user_thresholds consume exactly that, including ivs and onlytop AND-combination. What's missing is the *transport*: no JSON import path exists (IV1), add_threshold/share/edit cannot represent ivs (IV2 — an edit silently converts an explicit-IV target into match-everything floors), the loader has zero schema validation (IV5 — one malformed entry breaks the whole checker via TypeError at iv_checker.py:298, or silently no-ops via .get(..., 0) defaults), and league strings are normalized only by .capitalize() (IV6). Direction decision for Michael: teach _do_import to accept the JSON fragment format (sniff { → json path, else v1 text) + make ivs a first-class citizen of add/edit/share — or change the pogo-simulator emitter to the text format and give up explicit-IV cards. The first option is strictly more capable; the second loses the matchup cards entirely. Either way TS2's round-trip contract test should pin whichever shape wins.

(2) Option-2 migration readiness — MOSTLY READY ON IMPORT-PURITY, BLOCKED ON DATA-LAYER COUPLING. Verified clean: importing gopvpsim.user_collection triggers no network, no gamemaster load, and no numpy/numba (the heavy deps are not on that import path); packaging is a standard hatchling src-layout wheel. Blockers (CP9): gopvpsim's matcher hardwires its own network-backed data layer (separate cache dir, no bundled fallback, no ETag/conditional fetch — gobattlekit's fetcher is *ahead* here), and gopvpsim's pyproject hard-requires numpy+markdown, which a BeeWare iOS app must not drag in. Migration prep, in order: split gopvpsim extras (gopvpsim[sim] for numpy etc., core stays stdlib+certifi), inject the data layer (pass gamemaster/CPM providers into user_collection instead of importing its data module), then adopt. Behavioral deltas that must be reconciled *before* sharing code, because today the two apps give different answers for the same CSV row: shadow stat multipliers (CP3 — gopvpsim applies them, gobattlekit doesn't), over-cap exclusion (CP4 — gobattlekit is *right*: it starts the level walk at the mon's actual level so over-cap mons return None; gopvpsim reports them as qualifying), gender forms (CP5 — gopvpsim handles Oinkologne/Meowstic/Indeedee, gobattlekit's parser, matcher, and evo-line generator all lack gender), rank semantics (CP7 — competition-rank-over-floored-product vs dense-rank-over-float-with-tiebreak), IV/CP row validation (CP6), and CP-cap convention (CP8).

(3) CSV parsing parity — STRONG AGREEMENT ON THE PARSE, DIVERGENCE IN THE MATCH. The parity table (section F clean-areas) found the parsers row-for-row compatible on everything structural: DictReader by column name, utf-8-sig/BOM, byte-identical FORM_MAP dicts, shadow/lucky flags, level parsing including the same ValueError-skip for range strings, identical silent-skip malformed-row policy, identical CPM tables (101 entries), and equivalent stat-product formulas. The divergences are all in matching/validation semantics (CP3-CP8 above), not parsing — plus the genuine parse-level gap that gopvpsim reads a Gender column and gobattlekit doesn't (CP5). The branched pre-evo P0: gopvpsim solved it (reverse index + all-candidates loop + 8 tests, cycle-safe); gobattlekit's pre_evo_to_final overwrite remains, with blast radius mapped in DL9 (34 affected species, two screens; the generator and JSON are correct — the collapse happens in check_thresholds). SI5 — Android CSV import read loop likely writes NUL bytes: Python bytearray passed to Java InputStream.read() is converted by copy under Chaquopy NEEDS-DEVICE

likely-bug src/gobattlekit/screens/iv_checker.py:497-506 · confidence medium
