4 · Provenance UI design FOR YOUR REVIEW

Recommended: one merged list with per-target badges, colors, and ordering — not separate top-level sections. Why: provenance is per-target, not per-species (Florges already mixes sources across its targets); the IV checker is organized species-first; and user targets already live on their own screen ("My PvP IV Targets"), so the in-screen problem is only expert-vs-generated. Sectioning would double list length and force mixed species to appear twice.

  • Expert targets: teal (COLOR_ACCENT — the current matched-target color), unchanged look.
  • Generated targets: yellow (COLOR_YELLOW) + a constant SIM text badge — color alone isn't "super clear," so the badge always accompanies it. Expert targets always sort/cycle first.
  • A one-line source label under the target cycler ("Source: gobattlesim 2026-06 run (PvPoke-parity, no human review)").
  • Species whose targets are all generated get a [SIM] suffix in the species list.
  • Toggle: "SIM targets: shown/hidden" button at the top of the IV-checker species list, persisted via the existing preferences.py (show_generated_targets), enforced at the single chokepoint _get_thresholds so display, counts, and matching can never disagree. Renders only when generated targets exist — today's UI stays pixel-identical.

Schema: three optional per-target keys — class (expert|generated), source, desc — sibling to attack/defense/stamina/ivs/onlytop. Missing class ⇒ expert in shipped defaults, user in the user store. Verified mostly free: check_thresholds/qualifying_ivs ignore unknown keys; the one real break is user_thresholds._SPEC_KEYS rejecting unknown keys (must learn the three new ones), and edit/duplicate/share flows must carry the metadata so a generated import can't be laundered into an unlabeled target. Scope: ~200–300 lines across 7 files + ~100 lines of tests — one focused session.

Four decisions (each with my recommendation — I'll proceed with these unless you object)

Q1. Toggle default? Recommend: shown (badged). Hiding by default makes the pipeline's output indistinguishable from "no data for this species"; the badge carries the trust signal. This is the one most worth your veto — it's a brand/trust call about unsigned simulator output on first launch.
Q2. When a user imports a simulator paste into their own targets, does it keep the SIM badge? Recommend: yes, forever. Importing ≠ human validation.
Q3. Does hiding SIM targets also exclude them from matching? Recommend: yes. Counts always agree with what's displayed.
Q4. Per-target source strings from the pipeline, or one per species? Recommend: per-target. The exporter will emit them; species-level sources stays as fallback so iv_credits keeps working.

5 · Code reuse: artifact handoff + golden parity vectors

No import of gopvpsim into the app. The numpy hard-dependency (the old Option-2 blocker) stays out of the iOS build; the app consumes tiny data artifacts. The duplicated stat math (compute_rank_table, cp_to_level, shadow multipliers, CP caps, evolution lines) — which already drifted once and cost a batch of parity fixes — gets protected by automation instead: the export stage also emits golden parity vectors — (species, IVs, level) → (attack, defense, stamina, CP, rank) — for every shipped species, bundled as fixtures in gobattlekit's test suite. Drift becomes a failing test, not a wrong verdict in the field. The deeper fix (a numpy-free gopvpsim core the app imports) stays on pogo-simulator's backlog.
